# 07 - 记忆系统创新设计

> 本文档详细阐述 AI 桌面宠物记忆系统的创新设计理念、层级化分类体系、文件级与内容块级索引策略，以及 AI 自主记忆管理机制。该设计打破了传统 AI 应用中"记忆即数据库记录"的固有模式，实现了人类可读性与机器检索效率的统一。

---

## 一、核心理念：文件即记忆，数据库即索引

本项目的记忆系统打破了传统 AI 应用中"记忆即数据库记录"的固有模式，提出了 **"文件即记忆，数据库即索引"** 的全新设计理念。具体而言：

- **记忆本体**以人类可读的 **Markdown 文件** 形式持久化存储，用户可随时打开查看、编辑或备份，彻底消除 AI 记忆的"黑盒"属性，增强透明度和信任感
- **数据库（SQLite）** 仅作为高性能检索后端，为文件中的内容建立多维度索引，不承担记忆内容的存储职责
- 文件系统与数据库各司其职、协同工作，兼顾了**人类可读性**与**机器检索效率**的双重需求

> **创新点**：这种设计将记忆的所有权真正交还给用户——即使数据库完全损坏，原始的 Markdown 记忆文件依然完好无损，用户的记忆资产永不丢失。


### 二、记忆的层级化分类体系

记忆并非铁板一块，而是根据时间尺度与重要性呈现天然的层次结构。本项目将记忆划分为五个层级，形成一个从"易失"到"永恒"的完整光谱：

| 记忆层级 | 时间尺度 | 典型内容 | 管理策略 |
| :--- | :--- | :--- | :--- |
| **永久记忆** | 永久保留 | 用户姓名、生日、核心价值观、长期目标 | 人为审核确认，AI无权自动删除 |
| **长期记忆** | 数周至数年 | 重要事件回顾、阶段性总结、成长轨迹 | 由AI定期从短期记忆中提炼摘要生成 |
| **工作记忆** | 一次会话期间 | 当前正在进行的任务、待办事项、项目上下文 | 会话结束时决定保留、归档或丢弃 |
| **短期记忆** | 最近数轮对话 | 完整的对话历史、最近的交互内容 | 滑动窗口管理，超出长度自动淘汰 |
| **超短期记忆** | 数分钟之内 | 用户当前的情绪状态、最近的表情/语气变化 | 实时更新，快速遗忘，用于即时响应 |

> **创新点**：超短期记忆的设计尤其独特——它让宠物具备了"察言观色"的能力，能够感知用户情绪波动并及时调整回应策略，使交互更具"人情味"。


### 三、文件级索引策略

针对上述五类记忆，系统建立了精细化的文件组织与索引机制：

#### 3.1 文件目录结构

记忆文件按层级分类存储于不同子目录中，命名规范清晰：

```
data/memory/
├── permanent/          # 永久记忆
│   └── user_profile.md
├── long_term/          # 长期记忆
│   ├── 2024_q1_summary.md
│   └── 2024_q2_summary.md
├── working/            # 工作记忆
│   └── project_alpha_tasks.md
├── short_term/         # 短期记忆（按日归档）
│   ├── 2026-07-14_session.md
│   └── 2026-07-13_session.md
└── ultra_short/        # 超短期记忆（运行时临时）
    └── current_mood.md
```

#### 3.2 文件级索引表（`files`）

数据库中的 `files` 表为每个记忆文件建立索引条目，追踪文件的元信息变化：

```sql
CREATE TABLE files (
    id INTEGER PRIMARY KEY,
    pet_id INTEGER,              -- 支持多宠物实例隔离
    path TEXT,                   -- 文件相对路径
    category TEXT,               -- 记忆层级分类
    mtime INTEGER,               -- 文件修改时间戳（增量索引依据）
    hash TEXT,                   -- 内容哈希（去重与变更检测）
    UNIQUE(pet_id, path)
);
```

> **创新点**：通过 `mtime` 和 `hash` 双字段实现**增量索引**——系统仅当文件实际发生变化时才重新索引，极大节省了计算资源，使记忆系统的运行更加轻量高效。


### 四、内容块级索引策略

文件是粗粒度的存储单元，真正实现精准检索需要深入到内容片段层面。系统通过 `chunks` 表对每个 Markdown 文件进行细粒度切分与索引：

```sql
CREATE TABLE chunks (
    id INTEGER PRIMARY KEY,
    file_id INTEGER,             -- 关联所属文件
    category TEXT,               -- 继承文件的层级分类
    start_line INTEGER,          -- 内容在文件中的起始行
    end_line INTEGER,            -- 内容在文件中的结束行
    text TEXT,                   -- 记忆片段原文
    embedding TEXT,              -- 文本的向量表示（JSON格式存储）
    FOREIGN KEY (file_id) REFERENCES files(id)
);
```

基于此，系统支持**三种互补的检索路径**，确保任何场景下都能高效命中目标记忆：

1. **结构化查询（确定性查找）**：直接检索 `facts` 知识图谱表，精准回答"用户叫什么名字？""我的生日是哪天？"等事实性问题，速度最快、零误差
2. **关键词搜索（全文检索）**：利用 SQLite FTS5 扩展，支持中文分词与模糊匹配，适合"之前聊过关于旅游的什么内容？"等场景
3. **语义搜索（向量检索）**：通过 `embedding` 字段进行相似度计算，处理"那个让我印象很深的事情是什么来着？"等模糊联想式问题

三种检索方式可按优先级依次尝试或并行执行，最终由 AI 自主选择最匹配的结果。


### 五、AI自主记忆管理权

这是整个记忆系统最具突破性的设计——**记忆的管理权不再由开发者预设的算法掌控，而是由AI自身根据上下文自主决策**：

#### 5.1 写入自主权

AI 被赋予 `manage_memory` 工具，可在交互过程中自主判断何时需要写入记忆。典型触发场景包括：

- 用户明确表达了个人偏好或重要信息（"我最喜欢的颜色是蓝色"）
- 完成了一项值得记录的任务或里程碑
- 对话即将结束，需要对本次会话做摘要归档
- 检测到用户的长期兴趣或行为模式发生了显著变化

AI 调用工具时，以结构化参数指定写入内容与目标层级：

```json
{
    "category": "permanent",
    "content": "用户喜欢蓝色，偏好简洁风格的设计",
    "reason": "用户在对话中主动提及，且反复强调"
}
```

#### 5.2 索引维护权

系统在空闲时段（或由AI主动触发），会自动执行后台维护任务：

1. **摘要压缩**：将 `short_term` 中积累的对话内容提炼为结构化摘要，存入 `long_term`
2. **向量更新**：为新写入的内容生成 embedding 向量，更新 `chunks` 表
3. **事实抽取**：从对话中识别并提取"实体-属性-值"三元组，存入 `facts` 知识图谱
4. **冲突检测**：当新记忆与既有永久记忆冲突时，标记并请求用户确认

#### 5.3 查询策略自主权

当 AI 需要从记忆中检索信息时，它可以根据问题的性质自主选择最优检索策略：

- 事实性问题 → 优先查询 `facts` 结构化知识图谱
- 开放性回忆 → 启用语义向量搜索
- 关键词明确 → 使用 FTS5 全文检索
- 结果为空 → 自动降级到 Markdown 源文件全文扫描（最慢但最全）

> **创新点**：这套设计使AI从"被动接收指令的存储工具"进化为"主动管理知识的学习型智能体"，真正实现了记忆的自主化、个性化与智能化。


### 六、设计价值总结

| 维度 | 传统方案 | 本项目方案 |
| :--- | :--- | :--- |
| **数据所有权** | 数据存于专用数据库，用户难以直接查看 | Markdown 文件可读可编辑，所有权在用户手中 |
| **检索效率** | 依赖单一检索方式，场景适应性差 | 三层检索路径（结构化/关键词/语义）互为补充 |
| **上下文管理** | 简单窗口截断，丢失关键信息 | 五级分层体系，信息按重要性与时效性精细管理 |
| **AI自主性** | AI 仅读取预设的记忆内容 | AI 自主决定写入、摘要、检索策略，具备"元认知"能力 |
| **扩展性** | 记忆结构固定，难以适应新场景 | 文件 + 索引的松耦合设计，可灵活扩展新层级 |
| **容灾能力** | 数据库损坏则记忆全失 | 原始 Markdown 文件永远安全，重建索引即可恢复 |
